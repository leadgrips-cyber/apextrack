# ApexTrack Database Architecture

This document describes the PostgreSQL backend architecture for ApexTrack. It includes table relationships, core entities, and the ER diagram description for affiliate tracking, S2S postbacks, publisher approvals, and wallet-ledger support.

## Core Tables

- `publishers`: Stores affiliate partner profiles, approval state, assigned account manager, and operational status.
- `admins`: Stores back-office user accounts with RBAC roles for approval workflows and system administration.
- `offers`: Stores campaign definitions, tracking destinations, payout rules, geographic and device targeting, and approval requirements.
- `offer_applications`: Stores publisher requests to join or receive access to offers, enabling approval workflows.
- `clicks`: Stores tracked click events generated by publishers, including `sub` parameters, IP, browser, and metadata.
- `conversions`: Stores attributed conversion events linked to a click, with validation status, payout amounts, and S2S payload details.
- `wallets`: Stores wallet balances for each publisher, including available, pending, withdrawn, and hold balances.
- `wallet_transactions`: Stores ledger journal entries for wallet balance changes, referencing conversions and offers.
- `postbacks`: Stores server-to-server callback delivery state for conversions, including response tracking and retries.
- `api_tokens`: Stores API credentials for publishers and admins, with scopes, expiration, and active state.

## Relationship Summary

- `publishers` -> `wallets`: one-to-one relationship. Each publisher has one wallet.
- `publishers` -> `clicks`: one-to-many. Each click record is attributed to a publisher.
- `publishers` -> `conversions`: one-to-many. Each conversion record is attributed to a publisher.
- `publishers` -> `offer_applications`: one-to-many. Each publisher may apply to multiple offers.
- `publishers` -> `api_tokens`: one-to-many. Each publisher can own multiple API tokens.
- `admins` -> `offers`: optional creator association and permission records.
- `admins` -> `offer_applications`: reviewer association for approvals.
- `offers` -> `offer_applications`: one-to-many. Each offer receives publisher applications.
- `offers` -> `clicks`: one-to-many. Clicks are tracked per offer.
- `offers` -> `conversions`: one-to-many. Conversions are attributed to offers.
- `offers` -> `wallet_transactions`: one-to-many by reference when payout or adjustment is tied to an offer.
- `clicks` -> `conversions`: one-to-one. Each conversion references a click for attribution.
- `conversions` -> `postbacks`: one-to-many. A conversion may trigger zero or more S2S postbacks.
- `wallets` -> `wallet_transactions`: one-to-many. Ledger entries update wallet balances.
- `admins` -> `api_tokens`: one-to-many. Admins can also own API tokens.

## ER Diagram Description

The following describes the logical ER diagram in text form.

```
publishers ---< clicks
publishers ---< conversions
publishers ---< offer_applications
publishers --- wallet_transactions
publishers ---< api_tokens
publishers --- wallet (1:1)

admins ---< offers
admins ---< offer_applications
admins ---< api_tokens

offers ---< offer_applications
offers ---< clicks
offers ---< conversions
offers ---< wallet_transactions

clicks --- conversions (1:1)
conversions ---< postbacks
wallets ---< wallet_transactions
```

### Notes on important domain connections

- `offer_applications` provides the publisher approval workflow for access to restricted offers.
- `clicks` and `conversions` together support affiliate tracking and allow the system to trace a payout back to the originating publisher and offer.
- `postbacks` store the full S2S callback lifecycle so third-party networks or advertisers can receive conversion notifications and retry on failure.
- `wallet_transactions` act as the ledger, while `wallets` maintain current balances. Every balance update should be recorded in `wallet_transactions`.
- `api_tokens` support secure programmatic access, scoped separately for publishers and admins.

## Implementation Notes

- Uses `pgcrypto` for UUID generation with `gen_random_uuid()`.
- All tables include `created_at` and `updated_at` timestamps.
- `updated_at` is automatically refreshed via a shared trigger function.
- Strong indexes are defined for high-volume lookup paths: `clicks`, `conversions`, `postbacks`, `wallet_transactions`, and approval workflows.

## How this supports ApexTrack requirements

- Affiliate tracking: `clicks` and `conversions` both reference `publishers` and `offers`.
- S2S postbacks: `postbacks` stores callback payloads, statuses, and retry metadata.
- Publisher approvals: `offer_applications` captures requests and review state; `publishers.account_status` and `approval_status` track onboarding state.
- Wallet balances and ledger: `wallets` track balance snapshots while `wallet_transactions` provide the immutable transaction ledger.
- Admin control: `admins` and `api_tokens` allow operational users with distinct permissions.
- Production-ready: includes foreign keys, indexes, enum constraints, JSONB extension points, and explicit timestamp audit fields.
